# amx: deterministic session mechanics

Replace the claude-manager session's mechanical duties with a small
CLI, a zero-token hook, and thin judgment-only skills. The coordinator
workflow survives as an optional layer on the same state.

## Rationale

The manager skill implements an algorithm in prose: locking, registry
CRUD, watch loops, process-tree walks, tmux spawning. An LLM re-derives
it every session — slow, token-hungry, unreliable. Surveying the
ecosystem (claude-squad, uzi, agent-deck, Anthropic agent teams):
every working design keeps spawn/registry/status/cleanup in
deterministic code and reserves AI for judgment (briefs, journals,
review). The documented failures of AI-driven coordination (orphaned
sessions, stale status, managers needing prodding) match the pain that
motivated this redesign.

Key mechanic adopted from agent-deck: pre-generate the Claude session
id at spawn (`claude --session-id <uuid>`) and record it in the
registry at birth. This deletes the entire JSONL-hunting apparatus;
cold resume is `claude --resume <uuid>`.

## Naming

`amx` (agent multiplexer). Executes the rename direction in
`claude-manager-followups.md` § naming clash: nothing here is
Claude-specific and `claude-manager` is long and Claude-bound. The
acronym is overloaded elsewhere (Intel/Apple AMX, Harman AV) but
nothing claims the name on PATH, in brew, or locally.

- Binary: `amx`
- State dir: `~/.local/state/amx/`
- New skills: `amx-*`
- The `claude-manager-*` skill family keeps its names until the
  slimming pass (see Coordinator mode).

## Components

### amx CLI

Single shell script at `amx/amx` in this repo, symlinked to
`~/.local/bin/amx` by a Makefile target.

**Registry.** The existing markdown format, unchanged, at
`~/.local/state/amx/sessions.md`. Same `mkdir` lock,
re-read-under-lock discipline, field set, and lifecycle-by-field-
presence semantics as today — but implemented once, in the script.

**Source-of-truth rule: nothing edits the registry except `amx`.**
Skills, hooks, coordinator sessions, and workers all drive state
through `amx` commands. This is what keeps CLI days and coordinator
days on one system rather than two parallel ones.

**Commands (v1):**

- `amx spawn <name> [-r <repo>|-d <dir>] [-w] [-m <brief>]` —
  create tmux session `<name>` (detached, never steals focus), launch
  `claude --session-id <uuid>` in pane 0, write the registry entry
  with the id. Default directory is the cwd; `-r`/`-d` point
  elsewhere. `-w` creates a worktree branch via `wt` first (explicit
  opt-in; default is the repo/dir as given). `-m` passes the brief as
  Claude's positional initial-prompt argument — no send-keys, no
  TUI-readiness race. Branch naming and any ticket metadata are the
  session's business afterwards (`amx set`).
- `amx ls` — registry joined against `tmux ls`, one line per session:
  state (active/paused/shutdown), busy/idle for Claude panes (the
  existing capture-pane heuristics), drift flagged (entry without
  tmux, tmux without entry).
- `amx pause <name>` / `amx unpause <name>` — registry `paused` field
  plus the paused tmux option and switcher badge, as today (option
  renamed `@cm_paused` → `@amx_paused` at migration).
- `amx shutdown <name> | -a` — multi-window pane snapshot,
  resume_state file (per-pane cwd, command, claude session id via the
  registry — no JSONL hunt), registry rewrite, kill tmux (moving
  attached clients off first). Ported from the skill's § Shutdown.
- `amx resume <name>` — rebuild windows/panes/cwds from resume_state,
  replay recorded commands, `claude --resume <id>` for Claude panes.
  Best-effort by declared policy: a full dev-service rebuild may need
  hand-fixing; partial failure kills the half-built session and leaves
  the entry in shutdown (ported from § Cold resume).
- `amx wrap <name>` — remove the registry entry, kill tmux if alive.
  Journaling is the caller's job (see amx-wrap skill). No
  `wrap_requested` marker, no watch, no manager fulfilment phase.
- `amx set <name> <field> <value>` — registry field edit under the
  lock (ticket, notes, resume_target, …).

Deferred (v2 candidates): `amx send` / `amx output` (scripted
interaction with a session's Claude — useful for coordinator days),
switcher demotion of paused sessions.

### SessionEnd hook

`claude/hooks/amx-session-log.sh`, wired as a SessionEnd hook in
`settings.json.example`. Appends one JSON line to
`~/.local/state/amx/session-log.jsonl`: timestamp, session id, cwd,
git branch (if any), exit reason. Fires for every Claude session,
tracked or not — the safety net that lets an untracked session that
turned out to matter be retro-wrapped. Zero tokens, never blocks,
no LLM.

Readers dedupe by session id (resume cycles append multiple lines;
last line wins).

### Skills (judgment only)

- `amx-spawn` — natural-language layer when repo/scope needs figuring
  out; resolves the target and calls `amx spawn`. Secondary to the
  shell command.
- `amx-wrap` — run from inside a session: write the journal entry per
  the project's schema (the worker has the context the old
  manager-side wrap lacked), then call `amx wrap`. Wrap-time
  journaling is the norm.
- `amx-board` — on-demand cross-session review from any session:
  `amx ls`, stale-entry sweep (shutdown entries past their
  resume_target, unwrapped dead sessions in the stub log), journal/doc
  gap-fill. The coordinator's judgment as a query instead of a
  standing session. Also the home of the markdown task/agenda layer
  (`tasks.md` convention — see
  `2026-06-12-org-mode-assessment.md` § Direction).

Tracking is opt-in at spawn: ad-hoc `claude` runs get a stub-log line
and nothing else. No lifecycle obligations for throwaway sessions.

### Coordinator mode

The `claude-manager` skill family stays for deliberate
coordinator-session days, and adds the cross-session journal view when
a day warrants it. Follow-up (separate effort, after `amx` is proven):
slim those skills so all mechanics route through `amx` commands —
that pass also retires their prose mechanics, watch process, and
`wrap_requested` protocol, and is when the family rename to `amx-*`
roles happens.

## Migration

One-off, when the system is quiescent (no live manager, no in-flight
workers — the followups spec's live-migration caution):

1. Move `~/.local/state/claude-manager/` → `~/.local/state/amx/`.
2. Patch the state-dir path and `@cm_paused` option references in the
   `claude-manager-*` skills (mechanical; full rename deferred).
3. Sweep the existing shutdown ledger: wrap or drop each dead entry
   with the user.

## Out of scope

- `amx send` / `amx output`, switcher demotion (v2).
- Slimming/renaming the `claude-manager-*` skills (follow-up effort).
- vim org-mode integration and Apple-container dev sessions — separate
  proposals, specced independently.
